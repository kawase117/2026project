## 📋 前提

### ホール選びの基本原則

各店舗それぞれに、店内に多くの高設定を入れる日には異なった傾向がある。

唯一の例外として **マルハンメガシティ蒲田1と7** は共同でイベントをやっているが、細かい部分ではその中で違いがある。

蒲田1と7は以下のイベントを行う：
日曜日：機種1以上（それぞれの機種に一台は高設定が入る） 
土曜日：三台並び（台番号0001～0003のように連番で3台高設定が並ぶ） 
金曜日：列一台以上（0001～0010列に1台は必ず高設定が入る） 
木曜日：ニブイチ（複数機種の半分が高設定になる） 
水曜日：台番号末尾基準で高設定が入る 
火曜日；角◯（角からX番目の台に設定が入る） （最近は半分を高設定にして2末尾、ハーフという）
月曜日：一列まるごと高設定が入る（最近は列の半分を高設定にして2列、ハーフという）
また、日によって数機種全台に高設定を入れる傾向がある。

---

## 🎯 イベント属性の優先順位ルール

### 複合属性の優先付け

マルハンメガシティ蒲田1と7は、7のつく日や月末月初、不定期イベントなど**ほかのイベント要素が重なった際には曜日要素は無視される**。

**優先順位**（高低は曜日が一番下でそれ以外はものによる、要検証）:
1. **特別イベント** （演者来店、周年記念など）
2. **月末月初** （28日～2日）
3. **ゾロ目** （11日、22日、または強ゾロ目のMM=DD）
4. **7のつく日** （7日、17日、27日）
5. **曜日** （毎週の定期イベント）

## 📊 蒲田マルハン1＆7 の曜日イベント傾向

### イベント一覧

| 曜日      | イベント内容  | 設定投入パターン                    |
| ------- | ------- | --------------------------- |
| **日曜日** | 機種1以上   | 各機種に最低1台は高設定が入る             |
| **土曜日** | 三台並び    | 台番号0001～0003のように連番で3台高設定が並ぶ |
| **金曜日** | 列一台以上   | 0001～0010列に1台は必ず高設定が入る      |
| **木曜日** | ニブイチ    | 複数機種の半分が高設定になる              |
| **水曜日** | 台番号末尾基準 | 末尾の数字で判定（例：末尾3の台が高設定）       |
| **火曜日** | 角○      | 角からX番目の台に設定が入る              |
| **月曜日** | 一列まるごと  | 特定の列全体が高設定になる               |

### その他の店舗パターン

みとや大森町は **4のつく日** が優良など、店舗ごとに異なる傾向がある。

## 🏪 ホール選びの2つのアプローチ

### アプローチ1：全体成績が良い日を狙う

**判定方法**:
- 過去の属性が同じ日のホール全体での成績が良い
  - ↓
- 結論：ホール全体に高設定の台が多く入っている

**メリット**:
- ホール全体で設定が入っているため、狙い台がハズレても他の台にも期待できる
- 信頼度が高い

**デメリット**:
- 他の客も同じ情報を持っているため倍率が跳ね上がる
- 座りたい台に座れないリスクがある

---

### アプローチ2：特定の台が好調な日を狙う

**判定方法**:
- ホール全体の成績は悪くても、特定の台の成績が良い
  - ↓
- 結論：その台がなんらかの周期性で高設定を投入していると考える

**メリット**:
- 誰も気付いていない可能性が高い（倍率が低い）
- 確度が高い場合、確実に稼げる

**デメリット**:
- 1台に賭ける形になるため、ブレが大きい
- サンプル数が少ないと判定を誤る

## 🎯 ホール選びの基本戦略

### 理想的な選択

**理想は誰も居ない店で自分だけしか分からない高設定を狙うことだが理想論となる。**

### 現実的な対応

1. **基本：ホール全体の成績が良い日を狙う**
2. **倍率対策：複数の候補を用意する**
   - 店舗にして **3店舗**
   - 台数にして **10台** は欲しい

理由：倍率によっては狙い台に座れないことも考えられるため。

## 📍 ホール選びの要素

### ホール選びに影響する属性

#### 1. **店名（Hall Name）**
- 当たり前だが店によって資本力や客付きが異なる
- 高設定投入率が大きく異なる
- 投入傾向も変わる

#### 2. **日付（Date: YYYYMMDD）**

##### 年度的な傾向
- 年度末は弱い傾向
- ただし基本的にはDDと曜日で判断

##### 特別イベント
- **周年記念日**：どこの店も特に好成績 （優先度最高）

#### 3. **日（DD：1～31）**
*「日末尾」と区別するために「DD」と呼ぶ*

例：毎月10日は強い傾向

#### 4. **日末尾（Last Digit of Day）**
- DDの末尾を取ったもの
- 例：14日なら末尾は **4**、1日なら末尾は **1**
- 「4のつく日」など表現

#### 5. **ゾロ目パターン**

| パターン | 定義 | 例 |
|---------|------|-----|
| **ゾロ目** | 11日、22日 | 毎月同じ | 
| **強ゾロ目** | MMとDDが一致 | 12月12日、3月3日 |

#### 6. **曜日（Day of Week）**
- 毎週土曜日は好成績など
- 現在の地域では特にこれが強く出ている

### まとめ：属性の分析方向

これらを属性として過去成績を分析する。

## 🎲 台番号（台選び）の要素

### 台選びに影響する属性

#### 1. **台番号（Machine Number）**
- 座標、座席番号のようなもの
- 基本的には変化しない
- ただし工事やリニューアルで変化することもある

#### 2. **台番号末尾（Last Digit of Machine Number）**
- 台番号の最後の1桁
- ゾロ目の場合は2桁
- つまり台番号末尾は **0～9 + ゾロ目 = 11種類**

#### 3. **機種名（Machine Name）**
- 台の種類
- 映画館のAホールでハリー・ポッターの翌週にドラえもんが放映されるように定期的に変化

#### 4. **列（Column）** ⚠️ 別途位置データが必要
*位置データは現在準備中*

- 店によっては置いてある列丸ごと高設定などの設定投入をすることがある

#### 5. **角○（From Corner）** ⚠️ 別途位置データが必要
*位置データは現在準備中*

- 角から何番目に台が置かれているのか

### その他の台選び要素（今後追加予定）

- 前日比較
- 据え置き状況
- 移動平均
- など

**現在の方針**：ざっくり概要を掴むためにこの程度にしておく

## 📈 過去成績とは

### 成績を判定するための3指標

**G数、差玉、勝率** を基本として分析する。

#### 1. **G数（Games）**

**役割**：成績の信頼度を一定保証する

**考え方**：
```
例1：100G × 3000枚出 → まぐれ勝ち（その後やめられている）
例2：6000G × 2000枚出 → 高設定である可能性が高い
```

**重要な注意**：ギャンブルなのでブレがある
- 高設定でも負ける
- 低設定でも勝つ
- また、7000Gで-5000枚等は途中でやめない事情があった、高設定の負けと考えられるケースも多い

#### 2. **差玉（Difference Coins）**

**定義**：最終的に何枚のメダルが出たかを +-で表現

**目標値**：
- 一日の収支としては **＋2000枚（4万円）程度** は狙いたい
- ただしギャンブルなのでブレることが往々にしてある
- **差玉だけを考えるのは非常に危険**

#### 3. **勝率（Win Rate）**

**役割**：G数と同じくある程度の信頼度を担保する

---

## 🔍 特殊な状況での判定

### 極端なG数 × マイナス差玉のケース

```
極端に G数が大きく × 差玉が大きくマイナスの場合

「通常なら途中で諦めて回すのをやめるが、
 特定の理由で続けた = 高設定」

などの事情も考えられるため、有機的な判断が必要。
```

**つまり**：数値だけでなく、背景を考慮する必要がある

---

## ⚡ 倍率推定について（追加説明）

### G数ベースの倍率推定

倍率が低い候補は **G数である程度推測ができる**。

#### 倍率判定のロジック

| 平均G数          | 倍率評価         | 理由              |
| ------------- | ------------ | --------------- |
| **4000未満**    | **低（穴場）**    | 台が埋まっていない可能性が高い |
| **4000～5500** | **中程度**      | ある程度人気がある       |
| **5500以上**    | **高（避けるべき）** | 終日満員に近い状態       |

#### 満員判定の基準

```
G数が一定以上多い → 台が埋まり、離席や辞めること無く回されている

大体 5000後半のGがあれば → 終日満員と考えている
```

## 🗺️ ルーティング（日々の運用）

### 前日の準備段階

1. **前日までに**：当日の過去成績の良いホールと良い台を探す
2. **台をピンポイントで狙うよりも**：基本的にはホールを優先して考える

### ホール優先の理由

```
【釣り堀理論】
魚の多い釣り堀の方が魚の少ない釣り堀よりも釣りやすい

つまり：
- 狙い台を外しても、店内で他に優秀な台がある可能性が高い
- ホール全体に設定が入っていることが重要
```

**ただし**：客の数が多すぎれば事情は異なる

### 過去成績分析の流れ

#### ステップ1：属性別ホール成績の比較

```
登録してあるホール DBから、属性別の過去成績を比較する

↓ 必要な機能
「属性別の過去成績を表示するタブ」が必要

↓ 実行方法
属性別の過去成績を表示する（ホール全体）
```

**判定**：属性別の過去成績を比較する

**基準**：有機的な判断が必要だが、基本的に最も差玉が良い店を選ぶ

#### ステップ2：同様に台の属性別成績を比較

```
「属性別の過去成績を表示するタブ」で台単位の分析
```

**現在の状況**：クロス分析タブで行える

#### ステップ3：今後の自動化計画

```
ステップ1-2 を自動化して一括で表示できるようにする

↓ 自動化の条件
突発イベント（演者来店など）はないものとする
```

## 🤖 自動化の定義

### 「自動化」とは何か

```
現在あるクロス分析タブをホール向けにしたようなもので一覧表示すること
```

つまり：
- ホール選びに特化した、属性別の過去成績を自動抽出
- ユーザーが「選べる」形で複数の候補を一括表示
- 判断はユーザーが行う（自動推奨ではなく、材料提供）

### 自動化の手法

1. **属性が明確な日** （曜日、DD、末尾など）
   - ↓ 自動化できる
   - これらの属性で過去成績を集計

2. **突発イベント** （演者来店など）
   - ↓ 自動化の対象外
   - ユーザーが手動で判断する

## 📊 実装の方向性（概要）

### 第1段階：網羅的表示

```
【理想系】を選択肢の中から選ぶのではなく、
最初は網羅的に表現した中から選ぶ
```

つまり：
```
┌────────────────────────────────────────────┐
│ 複数の属性を一覧表示（曜日別、DD別、末尾別等）
│                                            │
│ ├─ 曜日別     (火曜日の過去成績)          │
│ ├─ DD別       (15日の過去成績)            │
│ ├─ 日末尾別   (末尾5の過去成績)           │
│ ├─ 全期間     (全体の平均値)              │
│ └─ カスタム   (任意の属性を指定)          │
│                                            │
└────────────────────────────────────────────┘

↓ ユーザーが「最適な属性」を選ぶ
↓ その属性でホール・台を比較表示
```

- 倍率スコアリング

## 📝 用語集

| 用語 | 説明 | 例 |
|------|------|-----|
| **DD** | 月内の日付（1～31） | DD=15 = 毎月15日 |
| **日末尾** | DDの末尾1桁 | 14日 → 末尾4 |
| **ゾロ目** | 11日、22日 | 変わらず毎月同じ日 |
| **強ゾロ目** | MM=DD | 12月12日、3月3日 |
| **倍率** | ホールの混雑度 | G数で推測 |
| **高設定** | 収支が好調な台 | 差玉＞0 or 勝率＞60% |
| **セッティング** | ホールが設定投入する行為 | 火曜日 = 角○ |
