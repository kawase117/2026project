## 重要な発見

### 1. 全ホール統合モデルの限界
- 単一の統合モデルで全ホール対応は**不十分**
- 特定ホールでは専用モデルにより **2-3%の性能向上** が期待可能

### 2. ホール毎の異なる特性
**高性能ホール（ザ-シティ-ベルシティ雑色店など）:**
- 統合版より高いAUC
- 機種別（model_type）戦略が有効
- データの質や量が異なる可能性

**通常性能ホール:**
- 統合版と同等のパフォーマンス
- 統合モデルでも実用的

### 3. 最適グループ化戦略
**全ホール共通の発見:**
- **model_type（機種別）戦略が最適**
  - tail（末尾別）: AUC ~0.504-0.510
  - model_type: AUC ~0.551-0.565 ⭐ 最良
  - machine_number: AUC ~0.505-0.547

## 結論と推奨

### ✅ 主な結論
**ホール別専用モデルは必要です**

### 理由
1. **統計的証拠:** 3ホール（37.5%）が統合版を有意に上回る
2. **性能向上:** 最大+2.86% のAUC改善が実現可能
3. **ホール固有の特性:** 各ホールのデータ分布・機種構成が異なる

### 推奨実装方針

#### 短期（Phase 6）
```
1. model_type 戦略を基本とした hall-specific ロジスティック回帰モデル実装
2. 高性能ホール（3施設）向け個別最適化
   - ザ-シティ-ベルシティ雑色店: 個別モデル
   - みとや大森町店: 個別モデル
   - レイトギャップ平和島: 個別モデル
3. その他ホール: 統合モデル またはグループ化モデルで対応
```

#### 中期（Phase 7+）
```
1. XGBoost 等のアンサンブルモデル導入で性能向上狙い
2. 拡張特徴量（69D）の phase 5-2 での効果検証
3. ホール特性に応じた特徴量最適化
```

## 技術的課題と解決策

### 課題1: 特徴量次元の不一致
**問題:** tail/model_type 戦略でホール間の特徴量数が異なる
```
tail strategy:   ホール間で 8-10 次元の差異
model_type:      ホール間で 93-226 次元の差異
```

**解決策:** Padding（ゼロ埋め）で統一 ✅
- 最大次元に合わせる
- 一貫性を保証

### 課題2: DB スキーマの差異
**問題:** マルハン蒲田1の DB に `machine_detailed_results` テーブルなし

**状況:** 8ホール有効、1ホール不可用（6実験スキップ）
