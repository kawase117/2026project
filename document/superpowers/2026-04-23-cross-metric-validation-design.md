# クロスメトリック検証（パーセンタイル比率自動最適化） 設計書

**作成日:** 2026-04-23  
**目的:** 複数の訓練指標（勝率・G数・差枚）から、テスト期間の差枚獲得を予測・検証するシステム設計

---

## 1. 背景と問題認識

### 現状の検証構造
現在のバックテスト検証は、以下の3つの相対パフォーマンス分析を並行実施：

- **相対パフォーマンス分析（勝率）** — 訓練期間の勝率でグループ分割 → テスト期間の勝率を検証
- **相対パフォーマンス分析（差枚）** — 訓練期間の差枚でグループ分割 → テスト期間の差枚を検証  
- **相対パフォーマンス分析（G数）** — 訓練期間のG数でグループ分割 → テスト期間のG数を検証

### 認識された課題

1. **単一メトリック循環** — 同じ指標で分割して同じ指標を検証するため、本質的な予測性（他の指標がどれだけ有効か）が不明確
2. **パーセンタイル比率の恣意性** — 現在の上位36%・中間28%・下位36%に理論的根拠がない
3. **複数訓練期間の統合方法が未定義** — 6月・3月・1月の3つの訓練期間結果をどう評価するか明確でない
4. **単日リスク** — 個別の大勝ちなど、外れ値の影響が見えない

### 新検証の目的

**クロスメトリック検証により：**
- 訓練期間の「勝率」がテスト期間の「差枚」を予測するか検証
- 訓練期間の「G数」がテスト期間の「差枚」を予測するか検証
- 各予測指標の「相対値」と「テスト期間での勝率」を並行測定し、外れ値パターンを発見

**パーセンタイル比率の自動最適化により：**
- 複数の比率（50-0-50、45-10-45、40-20-40、36-28-36、33-34-33）を自動試行
- 複数訓練期間での「勝者の一貫性」が最高且つ「相対値が最大」な比率を推奨
- テスト期間への過学習を避けつつ、最も堅牢な設定を発見

## 2. システムアーキテクチャ

### 2.1 検証構成（3層構造）

```
層1: 既存検証（保持）
├── relative_performance_multi_period_triple.py
│   └── 訓練勝率 → テスト勝率（法則性確認）
├── relative_performance_analysis_coin_diff_triple.py
│   └── 訓練差枚 → テスト差枚（基本検証）
└── relative_performance_analysis_games_triple.py
    └── 訓練G数 → テストG数（安定性確認）

層2: 新規クロスメトリック検証（本設計）
└── cross_metric_validation_triple.py（新規）
    ├── 訓練勝率グループ → テスト差枚 + テスト勝率
    ├── 訓練G数グループ  → テスト差枚 + テスト勝率
    └── 訓練差枚グループ → テスト差枚 + テスト勝率（参考）

層3: ウォークフォワード検証（未定義）
└── 後続フェーズで実装
```

### 2.2 パーセンタイル比率の自動最適化パイプライン

```
入力：
  - 訓練期間データ（6月、3月、1月）
  - テスト期間データ（4月1-20日）
  - 試行比率セット：[(50,0,50), (45,10,45), (40,20,40), (36,28,36), (33,34,33)]

処理フロー：
  1. 各比率について、3つの訓練期間で独立に：
     a. グループを分割（上位・中間・下位の割合は比率に従う）
     b. テスト期間での相対値と勝者を計算
  
  2. 3訓練期間の結果を集計：
     - 勝者の一貫性をチェック（3期間すべてで同じグループか）
     - 相対値の平均値を計算
     - 相対値の標準偏差を計算
  
  3. 推奨比率を決定：
     - 一貫性が高い（3期間で勝者が同じ）比率をフィルタ
     - その中から相対値が最大のものを選択
     - その比率に「推奨」フラグを付与

出力：
  - 全比率の比較表（一貫性・相対値・標準偏差を表示）
  - 推奨比率をハイライト
```

## 4. データフロー詳細

### 4.1 入力データ

```
machine_detailed_results テーブル：
├── date（YYYYMMDD）
├── machine_number
├── machine_name
├── last_digit
├── games_normalized（G数）

daily_hall_summary テーブル：
├── date
└── day_of_week
```

### 4.2 訓練期間・テスト期間

```
訓練期間（3パターン）：
├── 6月訓練（2025-10-01 ～ 2026-03-31）
├── 3月訓練（2026-01-01 ～ 2026-03-31）
└── 1月訓練（2026-03-01 ～ 2026-03-31）

テスト期間（固定）：
└── 4月（2026-04-01 ～ 2026-04-20）
```

### 4.3 処理フロー（単一比率での実行例：50-0-50）

```
Step 1: 訓練期間（6月）で勝率グループ分割
  Input: 2025-10-01 ～ 2026-03-31 のデータ
  Process:
    - 属性別（機械番号など）の勝率を計算
    - ソート → 上位50%と下位50%に分割
  Output: 上位グループの属性値リスト

Step 2: テスト期間でそのグループの差枚を測定
  Input: Step1の上位グループ属性値 + 2026-04-01 ～ 2026-04-20 データ
  Process:
    - 対象属性値のテスト期間での平均差枚を計算
    - テスト期間での勝率も計算
  Output: 
    {
      'test_avg_coin': 8.3,  # テスト差枚
      'test_win_rate': 0.42,  # テスト勝率
      'condition_avg_coin': -5.2,  # 条件全体の平均差枚
      'relative': 13.5  # 相対値
    }

Step 3: 訓練期間（3月）・（1月）でも同じ処理を繰り返し
  → 3つの結果を集計

Step 4: 一貫性をチェック
  6月：上位グループ勝利
  3月：上位グループ勝利
  1月：上位グループ勝利
  → 一貫性 ✅（3/3で同じ勝者）
  
Step 5: 相対値の統計
  相対値平均: (13.5 + 11.2 + 9.8) / 3 = 11.5
  相対値σ: 1.6
```

### 4.4 複数比率の結果統合（自動最適化）

```
試行結果表（DD別、機械番号属性、訓練勝率グループ分割）：
────────────────────────────────────────
比率      一貫性  相対値μ  相対値σ  推奨
────────────────────────────────────────
50-0-50   ✅     11.5    1.6
45-10-45  ✅     10.8    0.9   ← オススメ
40-20-40  ✅     10.2    0.7
36-28-36  ✅      9.8    0.8
33-34-33  ⚠️      8.5    3.2

判定ロジック：
1. 一貫性が高い（✅）を抽出 → 50-0-50, 45-10-45, 40-20-40, 36-28-36
2. その中から相対値が最大 → 50-0-50（11.5）
3. しかし「過学習リスク」を考慮し、次点の相対値が次候補…

実装上：
- 一貫性が高い中から「相対値が最大」を推奨
- ユーザーが目視で他の候補を選択可能
```

## 6. テスト戦略

### 6.1 単体テスト

- `analyze_cross_metric_validation_win_rate()` — 単一比率での分析が正しく計算されるか
- `find_optimal_percentile_ratio()` — 複数比率を試行して推奨比率が正しく選ばれるか
- `split_groups_triple_custom()` — 任意の比率でグループが正しく分割されるか

### 6.2 統合テスト

- クロスメトリック検証の全フロー（DBロード → 複数訓練期間処理 → 比率最適化 → 結果出力）
- 既存検証（相対パフォーマンス分析）との並行実行で矛盾がないか

### 6.3 検証テスト

- 出力結果が期待フォーマットと一致するか
- 一貫性判定ロジックが正しく機能するか
- 推奨比率がユーザーの判定基準（一貫性優先、相対値大）と一致するか

## 8. 設計上の決定事項

| 項目 | 決定内容 | 理由 |
|------|--------|------|
| **比率試行セット** | 50-0-50, 45-10-45, 40-20-40, 36-28-36, 33-34-33 | 2分割相当から3分割相当まで幅広くカバー |
| **推奨判定基準** | 一貫性✅ 中で相対値最大 | 過学習を避けつつ予測力を確保 |
| **訓練期間** | 6月・3月・1月（3パターン） | 異なる季節性の検証、複数期間の堅牢性確認 |
| **テスト期間** | 4月1-20日（固定） | 既存検証との統一 |
| **属性** | 機械番号・機種名・台末尾 | 既存検証と同様 |
| **条件分類** | DD別・曜日別 | 既存検証と同様 |
